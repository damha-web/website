# 서브 에이전트 설정 및 라우팅 전략 v2 (Sub-Agent Strategy)

본 문서는 서브 에이전트 오케스트레이션을 "운영 가능한 규칙"으로 정의한다.
목표는 라우팅 재현성, 품질 검증, 토큰 절약, 컨텍스트 안정성 확보다.

## 0. 적용 범위

- 본 문서는 이 프로젝트의 로컬 Override 규칙이다.
- 공통 운영 원칙은 Core 템플릿을 따르고, 프로젝트 특화 내용은 여기에서 관리한다.
- 세부 표준 문서는 아래 파일을 참조한다.
  - `docs/01-plan/subagent-routing-matrix.md`
  - `docs/01-plan/subagent-agent-contract-spec.md`
  - `docs/01-plan/subagent-failure-fallback-runbook.md`
  - `docs/01-plan/subagent-quality-gate-checklist.md`

## 1. 라우팅 대상 에이전트

### 1) App/UI Development (`stitch` + `bkit`)
- 트리거: UI/UX 설계, React/Next.js 컴포넌트 구현, 디자인 변형 생성
- 역할: 디자인 생성, 코드 변환, 스타일 가이드 정렬

### 2) Complex Tasks (`SequentialThinking`)
- 트리거: 아키텍처 설계, 다단계 추론, 전략 수립, 원인 분석
- 역할: 문제 분해, PDCA 기반 검증 루프, 의사결정 구조화

### 3) Browser Validation (`browser_subagent`)
- 트리거: 실제 브라우저 상호작용 및 동작 검증 필요
- 역할: 페이지 조작, DOM 확인, 검증 아티팩트(비디오/스크린샷) 확보

### 4) Knowledge & Context (`Knowledge` + `context7`)
- 트리거: 최신 라이브러리 문서 확인, 과거 작업 지식 재사용 필요
- 역할: 최신 공식 정보 확보, KI 요약 기반 중복 분석 최소화

### 5) External Ops (`github`, `TalkToFigma`, `notebook_mcp`, `everything`)
- 트리거: 형상관리 자동화, 피그마 매핑, 데이터 계산 검증, 세션 리소스 관리
- 역할: 외부 연동 실행과 증거 제출

## 2. 라우팅 규칙 (운영 기본)

- 복합 요청은 먼저 `SequentialThinking`으로 하위 작업으로 분해한다.
- 1차 실행에서는 최대 2개 에이전트만 활성화한다.
- 안전/권한 제약이 있는 작업은 기능 우선순위보다 권한 검증을 먼저 수행한다.
- 최신성 요구가 있는 기술 정보는 구현 전 `context7` 확인을 선행한다.
- 완료 보고는 설명 중심이 아니라 "변경 파일 + 검증 증거 + 리스크" 중심으로 작성한다.

## 3. 토큰 절약 및 컨텍스트 관리 규칙

- 턴 시작 고정 요약 3줄:
  - 현재 목표
  - 확정 제약
  - 완료 조건
- 턴 종료 고정 요약 3줄:
  - 결정사항
  - 미해결 리스크
  - 다음 액션 1개
- 긴 문서는 전체 재주입 대신 `TL;DR + 결정사항 + TODO`만 유지한다.
- 핸드오프는 계약 템플릿(목표/제약/입력/출력/검증)만 전달한다.
- 오래된 KI는 마지막 검증일과 신뢰도 태그가 없으면 참조하지 않는다.

## 4. 실패/폴백/중단 정책

- 재시도: 동일 작업 실패 시 에이전트별 최대 2회 재시도
- 폴백: 브라우저 자동화 실패 시 정적 분석 + 스크린샷 증거로 대체
- 중단: 보안, 권한, 정확성 임계치를 충족하지 못하면 사용자 승인 대기

세부 기준은 `docs/01-plan/subagent-failure-fallback-runbook.md`를 따른다.

## 5. 품질 게이트

모든 작업은 아래 4개를 충족해야 완료로 간주한다.

- 요구사항 충족 증거
- 회귀 위험 점검
- 문서/결정 로그 업데이트
- 검증 실행 결과 첨부

체크리스트는 `docs/01-plan/subagent-quality-gate-checklist.md`를 따른다.

## 6. 운영 변경 관리

- 전략 변경 시 본 문서 버전을 증가시키고 변경 이력을 남긴다.
- 라우팅 규칙이 바뀌면 매트릭스와 계약 문서를 동시에 갱신한다.
- 1주 운영 후 회고에서 KPI(1회 성공률, 재작업률, 평균 완료시간, 재요청률)를 점검한다.
