# 서브 에이전트 전략 실행 계획서 (Draft v1)

본 문서는 `subagent-strategy.md`를 운영 가능한 수준으로 구체화하기 위한 계획서다.
현재 단계의 목표는 "실행 전 검토 가능한 설계/운영 계획 확정"이며, 실제 설정 반영은 포함하지 않는다.

## 1) 문서 분석 요약

### 현재 문서의 강점
- 역할 기반 분리(UI, 복잡 추론, 브라우저, 지식, 외부연동)가 명확하다.
- 트리거와 도구를 함께 제시해 오케스트레이션 의도를 빠르게 이해할 수 있다.
- `DESIGN.md` 우선 확인, 품질 검증, 문서화 등 운영 원칙이 존재한다.

### 운영 관점의 공백 (실행 전에 보완 필요)
- 트리거가 정성적 표현 중심이라 라우팅 판단 기준이 주관적일 수 있다.
- 에이전트 간 우선순위/충돌 해소 규칙이 없다.
- 실패 시 폴백(fallback), 재시도, 타임아웃, 중단 조건이 없다.
- 입력/출력 계약(I/O contract)과 산출물 포맷이 미정이라 연계 자동화가 어렵다.
- 품질 지표(KPI/SLO), 평가 루브릭, 회귀 검증 절차가 없다.
- 지식(KI) 갱신 주기/신뢰도/충돌 해결 규칙이 없다.
- 보안/권한 경계(예: GitHub 쓰기 권한, 브라우저 민감정보 처리)가 문서화되어 있지 않다.

## 2) 목표 상태 (Target Operating Model)

### 핵심 목표
- 라우팅 정확도: 동일 유형 요청에서 일관된 서브 에이전트 선택
- 예측 가능성: 실패/지연/충돌 시 동일한 대응 경로 보장
- 검증 가능성: 모든 결과물이 품질 체크리스트를 통과해야 완료 처리
- 감사 가능성: 주요 결정(왜 이 에이전트를 선택했는지) 로그 보존

### 운영 원칙
- "최소 충분 도구" 원칙: 과도한 멀티툴 호출 방지
- "계약 기반 전달" 원칙: 에이전트 간 I/O 템플릿 강제
- "검증 우선" 원칙: 실행 결과가 없으면 완료로 간주하지 않음

## 3) 실행 범위와 비범위

### In Scope
- 서브 에이전트 라우팅 정책 정의
- 작업 분류 체계(Taxonomy)와 트리거 규칙 정량화
- 실패 대응 정책(재시도/폴백/중단) 정의
- 품질 게이트 및 문서 템플릿 정의
- 파일 기반 운영 가이드(런북) 초안 작성

### Out of Scope (이번 라운드)
- 신규 MCP 서버 개발/배포
- CI 인프라 대규모 개편
- 외부 시스템 권한 체계 재설계

## 4) 단계별 추진 계획

### Phase 0. 기준선 확정 (0.5일)
- 기존 전략 문서 버전 고정 및 변경 이력 섹션 추가
- 현재 워크플로우 샘플 5~10건 수집(최근 실제 작업)
- 성공/실패 사례를 기준선 데이터로 분류

산출물:
- Baseline 노트
- 작업 유형별 실제 사례 목록

완료 기준:
- "현재 방식으로도 충분한 영역"과 "개선 필요 영역"이 구분됨

### Phase 1. 라우팅 정책 설계 (1일)
- 작업 분류 체계 정의:
  - `ui_build`, `complex_reasoning`, `browser_validation`, `knowledge_refresh`, `external_ops`
- 요청 신호 기반 점수표 설계(키워드, 산출물 타입, 위험도)
- 충돌 해소 규칙 정의:
  - 1순위: 안전/권한 제약
  - 2순위: 산출물 요구 타입
  - 3순위: 비용/지연 최소화

산출물:
- Routing Matrix v1
- 우선순위 및 충돌 해결 규칙 문서

완료 기준:
- 동일 입력에 대해 라우팅 결과 재현성 확보

### Phase 2. 에이전트 계약(I/O) 표준화 (1일)
- 각 서브 에이전트 입력 템플릿 정의:
  - 목표, 제약, 컨텍스트, 성공 조건, 금지사항
- 출력 템플릿 정의:
  - 결과 요약, 변경 파일, 검증 결과, 리스크, 다음 액션
- 핸드오프 규약 정의:
  - `SequentialThinking -> stitch`, `Knowledge -> Complex Tasks` 등

산출물:
- Agent Contract Spec v1
- Hand-off Playbook

완료 기준:
- 핸드오프 시 누락 필드 0건

### Phase 3. 신뢰성 정책 (0.5일)
- 재시도 정책: 에이전트별 최대 횟수/백오프 정의
- 폴백 정책:
  - 예: `browser_subagent` 실패 시 정적 분석 + 스크린샷 증거 대체
- 중단 정책:
  - 권한/보안/정확성 임계치 미달 시 중단 및 사용자 확인

산출물:
- Failure & Fallback Runbook

완료 기준:
- 대표 실패 시나리오 5건에서 대응 경로가 문서로 완결됨

### Phase 4. 품질 게이트 및 관측 (1일)
- 공통 체크리스트:
  - 요구사항 충족, 회귀 위험, 문서 갱신, 검증 증거
- 지표 정의:
  - 라우팅 재작업률, 1회 성공률, 평균 완료 시간, 사용자 재요청률
- 결정 로그 템플릿 도입:
  - 선택 근거, 대안, 리스크, 후속 조치

산출물:
- Quality Gate Checklist
- Metrics Spec
- Decision Log Template

완료 기준:
- 샘플 작업 3건에 대해 지표 기록 가능

### Phase 5. 파일 반영 및 운영 시작 (실행 명령 이후)
- `subagent-strategy.md`를 운영 버전(v2)로 개정
- `docs/` 하위에 런북/템플릿 반영
- 시범 운영(1주) 후 회고 및 v2.1 조정

산출물:
- 운영 문서 세트 v2
- 1주 파일럿 회고 리포트

완료 기준:
- KPI 최소 2개 개선 또는 리스크 감소 근거 확보

## 5) 역할 및 의사결정 체계

- 사용자(Owner): 전략 승인, 우선순위 결정, 실행 승인
- 메인 에이전트(Orchestrator): 라우팅/검증/기록 책임
- 서브 에이전트(Executors): 도메인별 실행 및 증거 제출

승인 게이트:
- Gate A: 라우팅 정책 승인(Phase 1 종료 시)
- Gate B: 계약/실패정책 승인(Phase 3 종료 시)
- Gate C: 운영 반영 승인(Phase 5 시작 전, "실행해" 명령)

## 6) 즉시 적용 가능한 샘플 라우팅 규칙 (초안)

- 규칙 1: "UI 컴포넌트 구현 + 디자인 제약 명시" 요청은 `stitch + bkit` 우선
- 규칙 2: "전략/아키텍처/원인분석" 요청은 `SequentialThinking` 우선
- 규칙 3: "실제 브라우저 인터랙션 검증" 요구 시 `browser_subagent` 필수
- 규칙 4: "최신 문서 정확성" 요구 시 `context7` 조회 후 구현
- 규칙 5: "PR/브랜치/리뷰 자동화" 요구 시 `github` 연계
- 규칙 6: 복합 요청은 `Complex Tasks`로 분해 후 하위 라우팅

## 7) 리스크 및 선제 대응

- 리스크: 과도한 멀티 에이전트 호출로 지연 증가
  - 대응: 1차 라우팅에서 최대 2개 에이전트만 활성화
- 리스크: KI 오염(오래된 지식 재사용)
  - 대응: 신뢰도 태그 + 마지막 검증일 필드 강제
- 리스크: 브라우저 자동화 증거 부족
  - 대응: 비디오/스크린샷/DOM 로그 중 1개 이상 첨부 의무화
- 리스크: 품질 검증 생략
  - 대응: 완료 메시지 전 Quality Gate 체크 필수
