## 2) 보완 필요/권장 항목 (추가 확인 결과)

### A. 꼭 반영 권장 (실운영 안정성)

1. Webhook 응답 상태 검증
- `fetch`는 4xx/5xx에서도 예외를 던지지 않으므로 `response.ok` 체크 필요
- 실패 시 상태코드/응답본문을 서버 로그에 남겨 원인 추적 가능하게 처리

2. Webhook 호출 타임아웃
- 외부 API 지연으로 `/api/web-contact` 응답이 느려지지 않도록 `AbortController` 기반 타임아웃(예: 3초) 적용

3. 개인정보 로그 최소화
- 현재 route에서 전화번호 등 원문 로그를 남김
- 운영 로그에서는 전화번호 마스킹(예: `010-****-5678`) 권장

4. 메시지 길이 제한
- `message`가 매우 길 경우 알림 가독성과 전송 안정성 저하
- 서버에서 길이 상한(예: 1,000자) 적용 후 초과 시 잘라서 전송

### B. 상황 따라 권장 (스팸/운영 부담 완화)

1. 간단한 스팸 방어
- 최소 방어로 honeypot 필드 또는 요청 빈도 제한(간단 in-memory/edge rate limit) 고려

2. 실패 모니터링 기준
- 알림 실패가 사용자 제출을 막지는 않되, 장애 인지를 위해 `console.error` 패턴 통일

3. Block Kit는 2단계 적용
- 우선 `text` 기반으로 안정 배포
- 이후 테스트 성공 시 `blocks` 확장 적용 (미지원 환경 폴백 유지)

## 3) 승인 후 실행 계획 (상세)

## Phase 0. 사전 준비 (사용자)

1. 카카오워크 Incoming Webhook Bot 생성
2. 알림 채널 지정 후 Webhook URL 발급
3. 환경변수 등록
- 로컬: `.env.local`에 `KAKAOWORK_WEBHOOK_URL=...`
- Vercel: `Production` + `Preview`(필요 시) 동일 키 등록

## Phase 1. 서버 API 구현 (AI)

대상 파일: `src/app/api/web-contact/route.ts`

1. 기존 유효성 검증 로직 유지
- 필수값/전화번호 검증은 현재 구조 유지

2. 카카오워크 전송 함수 추가
- `sendKakaoWorkNotification(payload)` 형태로 분리
- 내부 처리:
  - 환경변수 없으면 조용히 skip
  - 타임아웃 적용된 `fetch`
  - `response.ok` 검사
  - 실패 로그 상세 기록(상태코드, 응답 텍스트 일부)

3. 실패 격리
- 알림 실패는 catch하여 기록만 하고, 사용자에는 기존 성공 응답 유지

4. 로그 정리
- 운영 로그는 개인정보 마스킹 후 기록

## Phase 2. 메시지 포맷 적용 (AI)

1. 1차 배포 포맷: `text` 단일 메시지
- 병원명/담당자/연락처/상품/문의내용 포함

2. 선택 적용: `blocks` 확장
- 테스트 후 적용 여부 결정
- 미지원/실패 시 자동 `text` 폴백 유지

## Phase 3. 검증 (AI + 사용자)

1. 정적 검증
- `npm run lint`

2. 기능 검증
- 정상 케이스: `/web` 폼 제출 → 카카오워크 알림 수신
- 환경변수 미설정: 폼 제출 성공 유지
- Webhook 오류 유도: 사용자 응답 성공 + 서버 오류 로그 확인

3. 배포 검증
- Vercel 재배포 후 프로덕션에서 1회 실전 전송 테스트

## Phase 4. 운영 전환

1. 운영 체크리스트 완료 후 적용 확정
2. Webhook URL 보안 관리(외부 공유 금지, 필요 시 재발급)
3. 장애 시 대응
- URL 오탈자/만료/권한 문제 우선 점검
